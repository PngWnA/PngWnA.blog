# 풀이
## 칼라 테이블
아무래도 LSB에 무언가를 숨겨놓았거나, 색 관련 데이터를 날린 것 같은데, 일단 이 당시에는 조금 더 아이디어를 얻기 위해서 [stegsolve](https://github.com/zardus/ctf-tools/tree/master/stegsolve)라는 프로그램을 사용해서 insight를 얻었습니다.

![](/images/DEFCON2019/misc_2_3.png)
gif파일의 칼라 테이블을 랜덤으로 매핑했을 때, 숨겨진 정보가 나타나네요. 도대체 왜?

[gif 헤더 구조](https://carstart.tistory.com/174)를 조금 찾아 본 후, 해당 gif의 색상 테이블쪽 헤더를 살펴봅시다.

![](/images/DEFCON2019/misc_2_4.png)


00 00 00, 00 00 00, 00 00 00, FF FF FF로 설정이 되어있네요. 아마 색상 테이블이 날라갔기 때문에 그림 자체가 제대로 나오지 않는 것 같습니다.

해당 테이블을 FF 00 00, 00 FF 00, 00 00 FF, FF FF FF로 수정하여 다시 gif파일을 열어봅시다.

![](/images/DEFCON2019/misc_2_5.gif)
잘 나오네요.

## 프레임 분석
gif를 보면 35개의 프레임이 저런식으로 각각의 도형으로 나오는데, 이제부터는 저 정보를 어떻게 해석할지의 문제입니다.

솔직히 여기서부터는 어느정도 직감이 들어갔습니다.

저 사진과 대회 룰로부터 우리가 참고할만한 내용은

1. 사진 아래를 보면 flag가 될 수 있는 문자 수가 정확히 32개입니다. 32는 2의 5승이니 5비트로 표현이 가능하겠네요.
2. 사진 아래를 보면 flag는 “OOO{“ 의 형태로 시작된다.

입니다.

flag가 32개니
* + : 0
* - : 1
…
* { : 30
* } : 31
의 매핑을 생각합시다. 딱 5비트로 flag 글자가 표현될 수 있다는 것이 우연은 아니라고 생각합니다.

## 프레임 해석
일단 gif를 보고 있으면 프레임마다 변하는 도형이 8각형 내부에서 왔다갔다 하는 것을 볼 수 있습니다.
첫번째로 생각해볼 수 있는 것은,

![](/images/DEFCON2019/misc_2_6.png)

처럼 각 꼭지점을 잡은 뒤, 해당 프레임의 도형 이 그 꼭지점을 포함하면 1, 아니면 0으로 표현하는 방법입니다.
해당 프레임은 11010111이 되겠군요.

1번째 꼭지점을 어디로 잡느냐가 문제가 될 수 있으나, 일단 첫번째로 시도할 때는 저 점을 1번째 꼭지점을 잡았습니다.
그러면 1, 2, 3번째 프레임을 비트로 표현해 봅시다.

10001100
01100011
11100100
…

규칙이 없는 듯 보입니다.
(라고 생각하고 잠시 10분동안 쉬었습니다…)

아까 flag 글자들이 5비트로 표현 가능하다고 했던 적이 있네요.
5비트씩 끊어봅시다.

**10001 (17='O')**
**10001 (17='O')**
**10001 (17='O')**
**11110 (30='{')**
0100 …

17, 17, 17, 30은 아까 숫자와 flag글자들의 매핑을 찾아보면
“OOO{“라는 글자가 나옵니다.
제대로 갈피를 잡은 것 같군요.
프레임도 35프레임이라, 5로 나눠떨어지니, 끝 글자까지 문제 없이 잘 나올겁니다.
다만 주의할 것이 있다면, 8각형이 프레임이 지나가면서 반시계방향으로 약간씩 회전한다는 것입니다.
이 점을 참고한다면, 수동으로 한 비트씩 변환하거나, 이미지 관련 라이브러리를 써서 자동으로 비트값을 추출하거나 해서 flag를 얻을 수 있습니다.
(문제를 풀 당시 새벽이라 툴 짜기 귀찮아서 수동으로 보면서 한 비트씩 변환했습니다… 프레임 수가 적어서 그렇지, 프레임 수가 많은 경우에는 이미지 관련 라이브러리를 써서 비트를 추출해야 할 것입니다.)

**flag : OOO{FORCES-GOVERN+TUBE+FRUIT_GROUP=FALLREMEMBER_WEATHER}**

# 참고할 내용

이런 스테가노그래피 냄새가 나는 문제들은 여러가지 살펴봐야 할 점이 있지만, 문제가 이미지로 나온 경우에는 hxd로 훑어볼 때 다음과 같은 부분을 살펴보면 좋습니다.

* 이미지 파일인데 뒤에 이상한 데이터가 붙어있나?
JPEG는 0xFFD9, PNG는 Ascii로 읽으면 “IEND” 같은 데이터가 이미지 파일의 끝에 붙습니다. 이러한 데이터로 끝나지 않는다면 뒤에 무언가 있지 않는가 생각하고 넘어가면 의문의 파일을 발견할 수 있을지도 모릅니다.

* 이미지를 열어봤는데 파일 용량이 보이는 이미지보다 비정상적으로 큰가?
이미지의 모서리쪽에 flag나 중요한 정보를 적어놓고 안보이게 만들기 위하여 이미지 파일의 헤더, 특히 넓이와 높이에 관한 부분을 조작하면, 아래쪽에 이미지가 있어도 뷰어에서는 안보이게 만들 수 있습니다.

* 만약 둘 다 아니라면?
이제부터는 생각을 좀 해봐야 합니다. flag를 이미지에 써놓고 배경색과 아주 미묘한 색 차이가 나게 글씨색을 바꿨을 수도 있고(이런 경우에는 포토샵 같은 프로그램으로 그 글시를 보려고 노력하거나, 프로그래밍을 통하여 직접 추출해야 합니다), 아니면 특정 색 채널에 비트정보를 숨겨놨을 수도 있습니다.
아니면 헤더에서 색 관련 내용을 이상하게 조작했거나, 날려버려서 원래 이미지가 제대로 표현되지 않게 만드는 경우도 있습니다.

이 문제는 마지막 경우였군요.

해당 내용들을 자세히 정리한 [블로그](https://bpsecblog.wordpress.com/2016/08/21/amalmot_4/)가 있습니다. 이것도 참고해보세요.
